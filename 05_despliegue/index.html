    Unidad 5 – Despliegue de aplicaciones Python con Docker

    1. Introducción al despliegue de aplicaciones

    Hasta ahora las aplicaciones Python se han ejecutado en local, utilizando entornos virtuales y bases de datos SQLite.
    En esta unidad se aprenderá a desplegar una aplicación Python de forma reproducible y portable mediante contenedores Docker.

    El objetivo de la unidad es:

    • Contenerizar una aplicación FastAPI
    • Utilizar una base de datos relacional PostgreSQL
    • Conectar la aplicación con la base de datos
    • Garantizar la persistencia de los datos
    • Comprender el flujo real de despliegue de una aplicación backend

    2. Qué es Docker y por qué se utiliza

    Docker es una plataforma que permite ejecutar aplicaciones dentro de contenedores.

    Un contenedor:

    • Incluye la aplicación
    • Incluye todas sus dependencias
    • Se ejecuta igual en cualquier entorno

    Ventajas principales:

    • Portabilidad
    • Reproducibilidad
    • Aislamiento de servicios
    • Facilidad de despliegue

    3. Arquitectura de la aplicación

    La aplicación que se va a desplegar consta de dos servicios:

    • Servicio 1: Aplicación Python con FastAPI
    • Servicio 2: Base de datos PostgreSQL

    Cada servicio se ejecutará en su propio contenedor, pero estarán conectados entre sí.

    4. Contenerización de la aplicación FastAPI

    4.1 Archivo requirements.txt

    La aplicación debe listar todas sus dependencias en un archivo requirements.txt.

    Ejemplo:

    fastapi
uvicorn
sqlalchemy
psycopg2-binary

    4.2 Creación del Dockerfile

    El Dockerfile define cómo se construye la imagen de la aplicación.

    Ejemplo de Dockerfile:

    FROM python:3.12-slim

WORKDIR /app

COPY requirements.txt .

RUN pip install --no-cache-dir -r requirements.txt

COPY . .

CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]

    5. Base de datos PostgreSQL en Docker

    PostgreSQL es una base de datos relacional, al igual que SQLite, pero orientada a entornos reales.

    Diferencias principales:

    • PostgreSQL funciona como servicio independiente
    • SQLite es un archivo local
    • PostgreSQL soporta concurrencia real y escalabilidad

    6. Persistencia de datos con volúmenes en Docker

    6.1 El problema de los contenedores

    Los contenedores Docker son efímeros.

    Si un contenedor se elimina:

    • Se pierde su sistema de archivos
    • Se pierden los datos de la base de datos

    6.2 Qué es un volumen Docker

    Un volumen Docker permite almacenar datos fuera del contenedor.

    Características:

    • Persisten aunque el contenedor se borre
    • Son gestionados por Docker
    • Se pueden reutilizar

    Idea clave:

    El contenedor es temporal,
    el volumen es persistente.

    6.3 Volúmenes y PostgreSQL

    PostgreSQL almacena sus datos en:

    /var/lib/postgresql/data

    Este directorio debe asociarse a un volumen para evitar la pérdida de datos.

    6.4 Definición y uso de volúmenes

    Definición del volumen:

    volumes:
  postgres_data:

    Asociación al contenedor:

    volumes:
  - postgres_data:/var/lib/postgresql/data

    7. Docker Compose: orquestación de servicios

    7.1 Qué es Docker Compose

    Docker Compose es una herramienta que permite definir y ejecutar múltiples contenedores de forma conjunta mediante un único archivo.

    Este archivo se llama docker-compose.yml.

    Docker Compose permite:

    • Arrancar varios servicios a la vez
    • Definir redes automáticamente
    • Gestionar volúmenes
    • Establecer dependencias entre servicios

    7.2 Estructura básica de docker-compose.yml

    Un archivo docker-compose.yml se estructura en secciones.

    Ejemplo básico:

    services:
  api:
    build: .
  db:
    image: postgres:15

    7.3 Definición del servicio de la API

    Ejemplo de servicio para la aplicación FastAPI:

    services:
  api:
    build: .
    ports:
      - "8000:8000"
    depends_on:
      - db
    environment:
      DATABASE_URL: postgresql://app:app123@db:5432/appdb

    Esto indica:

    • Se construye la imagen desde el Dockerfile
    • Se expone el puerto 8000
    • Depende del servicio de base de datos
    • Se define la cadena de conexión

    7.4 Definición del servicio de base de datos

    Ejemplo de servicio PostgreSQL:

    services:
  db:
    image: postgres:15
    environment:
      POSTGRES_USER: app
      POSTGRES_PASSWORD: app123
      POSTGRES_DB: appdb
    volumes:
      - postgres_data:/var/lib/postgresql/data

    7.5 Archivo docker-compose.yml completo

    services:
  api:
    build: .
    ports:
      - "8000:8000"
    depends_on:
      - db
    environment:
      DATABASE_URL: postgresql://app:app123@db:5432/appdb

  db:
    image: postgres:15
    environment:
      POSTGRES_USER: app
      POSTGRES_PASSWORD: app123
      POSTGRES_DB: appdb
    volumes:
      - postgres_data:/var/lib/postgresql/data

volumes:
  postgres_data:

    7.6 Comandos básicos de Docker Compose

    Arrancar servicios:

    docker compose up

    Arrancar en segundo plano:

    docker compose up -d

    Detener servicios:

    docker compose down

    Detener y borrar volúmenes:

    docker compose down -v

    8. Creación de tablas en la base de datos

    Docker y PostgreSQL no crean las tablas.

    Las tablas las crea la aplicación Python, usando los modelos.

    Ejemplo en main.py:

    from database.database import engine, Base
from models.user import User

Base.metadata.create_all(bind=engine)

    9. Flujo completo de despliegue

    1. Docker construye las imágenes
    2. Docker Compose crea la red y los contenedores
    3. PostgreSQL crea la base de datos vacía
    4. La aplicación se conecta a PostgreSQL
    5. La aplicación crea las tablas
    6. Los datos se guardan en el volumen

    10. Conclusión

    Docker y Docker Compose permiten desplegar aplicaciones Python de forma profesional.
